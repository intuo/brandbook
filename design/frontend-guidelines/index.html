Frontend guidelines

Bootstrap is a comprihensive and useful SCSS framework, which we use as our main foundation. But because of Bootstrap's overall large size, it's important to maintain an consistent project structure in order to the frontend development running smoothly. Our frontend guidelines are primarly based on the SCSS guidelines created by @mdo at Github.

HTML architecture

General formatting
  • Use soft-tabs with a two space indent. Spaces are the only way to guarantee code renders the same in any personâ€™s environment.
  • Paragraphs of text should always be placed in a <p> tag. Never use multiple <br> tags.
  • Every form input that has text attached should utilize a <label> tag. Especially radio or checkbox elements.
  • Even though quotes around attributes is optional, always put quotes around attributes for readability.
  • Avoid trailing slashes in self-closing elements. For example, <br>, <hr>, <img>, and <input>.
Slimming down elements

Whenever possible, avoid superfluous parent elements when writing HTML. Many times this requires iteration and refactoring, but produces less HTML. For example: 

  <!-- Not so great -->
  <span class="avatar">
    <img src="...">
  </span>

  <!-- Better -->
  <img class="avatar" src="...">
Boolean attributes

Many attributes donâ€™t require a value to be set, like disabled or checked, so donâ€™t set them. 

  <input type="text" disabled>

  <input type="checkbox" value="1" checked>

  <select>
    <option value="1" selected>1</option>
  </select>

What and why?

The Brandbook is Intuo's internal design reference. It describes how or when you should choose on of the many design patterns to ensure consistency.

There's a lot we can improve upon and feedback is always welcome. The Brandbook is an ongoing project that evolves naturally with time, hence we will occasionally add or remove things.

Thinking in components

We believe that every design component should be re-usable, or at least carefully documented what it's specific purpose is. Rather than introducing new components, try to figure out what problem you're solving, and how it can be solved using existing patterns.

Scalable and modular SCSS

Spacing
  • Use soft-tabs with a two space indent. Spaces are the only way to guarantee code renders the same in any personâ€™s environment.
  • Put spaces after : in property declarations.
  • Put spaces before { in rule declarations.
  • Put line breaks between rulesets.
  • When grouping selectors, keep individual selectors to a single line.
  • Place closing braces of declaration blocks on a new line.
  • Each declaration should appear on its own line for more accurate error reporting.
Formatting
  • Use hex color codes #000 unless using rgba() in raw CSS (SCSSâ€™ rgba() function is overloaded to accept hex colors as a param, e.g., rgba(#000, .5)).
  • Use // for comment blocks (instead of /* */).
  • Avoid specifying units for zero values, e.g., margin: 0; instead of margin: 0px;.
  • Strive to limit use of shorthand declarations to instances where you must explicitly set all the available values.
Misc

As a rule of thumb, avoid unnecessary nesting in SCSS. At most, aim for three levels. If you cannot help it, step back and rethink your overall strategy (either the specificity needed, or the layout of the nesting).

Examples

Here are some good examples that apply the above guidelines: 

  // Example of good basic formatting practices
  .styleguide-format {
    color: #000;
    background-color: rgba(0, 0, 0, .5);
    border: 1px solid #0f0;
  }

  // Example of individual selectors getting their own lines (for error reporting)
  .multiple,
  .classes,
  .get-new-lines {
    display: block;
  }

  // Avoid unnecessary shorthand declarations
  .not-so-good {
    margin: 0 0 20px;
  }
  .good {
    margin-bottom: 20px;
  }
Class name conventions

Never reference js- prefixed class names from CSS files. js- are used exclusively from JS files. Use the is- prefix for state rules that are shared between CSS and JS.

Specificity

When styling a component, start with an element + class namespace (prefer class names over ids), prefer direct descendant selectors by default, and use as little specificity as possible. Here is a good example: 

  <ul class="category-list">
    <li class="item">Category 1</li>
    <li class="item">Category 2</li>
    <li class="item">Category 3</li>
  </ul>

  .category-list { // element + class namespace

    // Direct descendant selector > for list items
    > li {
      list-style-type: disc;
    }

    // Minimal specificity for all links
    a {
      color: #f00;
    }
  }
CSS Specificity guidelines
  • When modifying an existing element for a specific use, try to use specific class names. Instead of .listings-layout.bigger use rules like .listings-layout.listings-bigger. Think about ack/greping your code in the future.
  • The class names disabled, mousedown, danger, hover, selected, and active should always be namespaced by a class (button.selected is a good example).
File organization

We're using a SCSS module based file organisation, to keep our files tidy. The base directory contains inherited bootstrap classes, the components directory contains custom Intuo modules. The layout directory contains files that only affects the application's strucutre.

Brandbook dependencies

  • Our styles are built with SCSS.
  • Middleman is used to serve the GitHub pages.
  • middleman-syntax for code highlighting.
  • turbolinks for rapid page loading.
  • We use our own Intuo icon font alongside of Ionicons.
  • Intuo's design framework is built on top of Bootstrap Sass 3.
  • Autoprefixer is used to generate vendor-prefixed declarations.
